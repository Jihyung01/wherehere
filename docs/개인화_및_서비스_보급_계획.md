# 개인화 추천 · 서비스 보급 계획

## 1. 사용자 정보를 “백엔드에 누적”하는 구조

### 1.1 지금 이미 되는 것

- **로그인**: 카카오/구글 → Supabase Auth → `user.id` (UUID) 발급
- **방문 기록**: 퀘스트 완료 시 `POST /api/v1/visits` → `user_id` = Supabase `user.id` 로 **visits** 테이블에 누적
- **추천 요청**: `POST /api/v1/recommendations` 에 `role_type`, `current_location`, `mood` 전달 → 이 값들로 장소 추천

즉, **누적**은 Supabase(및 연결된 DB)에 되고, 백엔드는 그 데이터를 읽어서 쓰는 구조입니다.  
“백엔드로 누적” = **백엔드가 접근하는 DB( Supabase )에 사용자별 데이터가 쌓인다**고 보면 됩니다.

### 1.2 더 쌓이면 좋은 데이터 (개인화 강화)

| 데이터 | 저장 위치 | 용도 |
|--------|-----------|------|
| **방문 기록** (이미 있음) | `visits` | 방문 패턴, 선호 카테고리, 누적 거리, 탐험 반경 |
| **역할/닉네임** | 온보딩 → Supabase `user_metadata` 또는 `profiles` | 역할 기반 추천, UI 표시 |
| **선호 카테고리** | `profiles` 또는 방문 기반 집계 | “이 사용자에게 카페/맛집 비중 높이기” |
| **평균 체류시간·비용** | `visits`에서 집계 | “비슷한 스타일 장소” 추천 |
| **거부/스킵 이력** (선택) | 새 테이블 `feedback` | “이런 장소는 덜 보여주기” |

지금도 **방문 + 역할 + 기분**만으로 어느 정도 개인화가 되고,  
위 항목들을 조금씩 추가하면 “그 사용자에게 더 색다르고 맞춤인” 추천으로 확장할 수 있습니다.

## 2. 개인화를 “서비스”로 만드는 단계

### 2.1 1단계: 로그인과 데이터 일관되게 쓰기 (현재 단계)

- 카카오/구글 로그인으로 **동일한 user.id** 유지
- 추천·방문·나의 지도 모두 **같은 user_id** 사용
- 온보딩에서 고른 **역할**을 Supabase `user_metadata` 또는 `profiles`에 저장해 두고, 추천 API 호출 시 `role_type`으로 넘기기

→ 이미 구조는 갖춰져 있으므로, **Supabase Provider 설정만 하면** “로그인한 사용자 기준으로 누적”이 됩니다.

### 2.2 2단계: 추천 로직을 “사용자별”로 강화

- **방문 기록**으로 “자주 가는 카테고리·지역” 계산
- **역할 + 기분**은 지금처럼 요청마다 넘기고,  
  **과거 방문**은 백엔드에서 `user_id`로 조회해 스코어에 반영
- 예: “이 사용자는 카페/맛집 비중이 높다” → 같은 역할이어도 카페/맛집 비중을 더 주는 식

이미 `get_user_visits` 등이 있으므로, **추천 API 내부에서**  
`user_id`가 있으면 방문 기반 선호도를 섞어 주도록만 확장하면 됩니다.

### 2.3 3단계: “색다른” 경험 강화

- **히든젬·덜 알려진 장소** 비중 조절 (역할/레벨에 따라)
- **이번 달 탐험 반경·누적 거리**를 활용해 “조금 더 먼 새 구역” 제안
- **요일/시간대**에 따른 추천 (저장된 선호 시간 또는 방문 패턴 기반)

→ 모두 **이미 쌓는/쓸 수 있는 데이터**를 조합하는 수준입니다.

### 2.4 4단계: 서비스로 “보급”할 때

- **실데이터만 사용**: Mock 제거, Supabase( places + visits ) + 수집 스크립트로 장소 확보
- **성능**: 추천/방문 조회에 인덱스·캐시(예: Redis) 도입
- **모니터링**: 로그인 실패율, 추천 클릭률, 방문 완료율 등 지표 수집
- **점진적 롤아웃**: 특정 지역/초대 사용자만 먼저 → 피드백 반영 후 전면 오픈

## 3. 요약: 질문에 대한 답

- **카카오/구글 로그인 구현**  
  - 프론트: 이미 버튼·OAuth 플로우·콜백(`/auth/callback`) 구현됨.  
  - 남은 작업: Supabase에서 Google/Kakao Provider 설정 + Redirect URL 등록 (위 설정 가이드 참고).

- **사용자 정보를 백엔드(DB)로 누적**  
  - 예. 로그인한 `user.id` 기준으로 **visits**에 누적되고, 추천 시 `user_id`를 쓰면 “그 사용자에 맞춘” 데이터로 확장 가능.

- **개인화·색다른 추천**  
  - 역할·기분(이미) + 방문 기록·선호 카테고리·탐험 반경 등을 DB에서 읽어 추천 로직에 반영하면, 사용자별로 더 맞고 색다른 추천이 가능.

- **서비스로 보급**  
  - 실데이터 전용 전환 → 개인화 로직 강화 → 성능/모니터링 → 소규모 테스트 후 전면 오픈 순서로 진행하면 됩니다.

설정만 완료하면 카카오/구글 로그인과 “사용자 정보 누적 → 개인화” 구조는 이미 갖춰져 있고, 위 계획대로 단계만 확장하면 됩니다.
