# WhereHere 추천 로직 분석 (거리 / 점수 / 역할 / 기분)

## 1. 현재 어떤 API를 쓰나요?

- **프론트**: `getRecommendations(lat, lng, roleType, mood)` → `POST /api/v1/recommendations`
- **바디**: `role_type`, `current_location`, `mood: { mood_text, intensity }`
- **백엔드**: `backend/routes/recommendations.py` → DB 연결 시 REST API로 장소 조회, 실패 시 Mock

---

## 2. 거리 반영 여부 (왜 먼 장소가 나오나)

### DB 경로 (실제 배포 시)

| 단계 | 코드 위치 | 현재 동작 |
|------|-----------|-----------|
| 1. 반경 조회 | `recommendations.py` L92–103 | 역할별 `radius_map`으로 반경 결정 (예: explorer 3000m, healer 800m) |
| 2. 장소 조회 | `db/rest_helpers.py` `get_places_nearby()` | **문제**: Supabase REST에 `radius_meters`를 넘기지만, **실제 쿼리에는 반경/거리 조건이 없음**. `is_active=true`, `limit=50`만 적용 → **전국(또는 DB 전체) 장소 50개** 가져옴 |
| 3. 3개 선택 | `recommendations.py` L126 | `random.sample(places, min(3, len(places)))` → **점수/거리 무관하게 랜덤 3개** 선택 |
| 4. 거리 계산 | L117–124, L149–161 | 선택된 3개에 대해 Haversine으로 `distance_meters` 계산해서 **표시용**으로만 사용 |

**정리**:  
- 거리는 **계산해서 화면에만 보여주고**,  
- **어떤 장소를 후보로 가져올 때(반경 필터)** 와 **그중에서 뭘 3개 뽑을 때(정렬)** 에는 **전혀 반영되지 않습니다.**  
그래서 457m / 4.2km / 9.8km 같이 먼 곳이 섞여 나옵니다.

### Mock 경로 (DB 실패 시)

- `mock_data.py`에서 Haversine으로 거리 계산 후  
  `dist_score = exp(-0.0003 * dist)` 로 **점수에 반영**하고,  
  `scored.sort(key=final_score, reverse=True)` 후 **상위 3개** 반환.
- Mock에서는 **거리가 점수에 반영되고, 점수 순으로 3개**가 나옵니다.

## 3. 점수(Score) 기준 — 어떻게 나오나

### DB 경로 (지금 쓰는 공식)

```text
score = 85.0
       + (average_rating × 2)        // 평점 4.5 → +9
       + max(0, 10 - distance_km)   // distance_km = distance_meters / 1000
```

- `score_breakdown`에 들어가는 값:
  - `category`: 고정 **40.0** (역할/카테고리 미반영)
  - `distance`: `max(0, 25 - distance_meters/200)` (표시용, 실제 선택 로직에는 미사용)
  - `rating`: `average_rating × 4`

예: 평점 4, 거리 457m  
→ 85 + 8 + (10 - 0.457) ≈ **102.5** (스크린샷 102.5와 일치)

즉, **점수는 “계산은 되지만”, 3개를 고를 때는 사용하지 않고 랜덤으로 고릅니다.**

### Mock 경로

```text
final_score = (cat_score × 40) + (dist_score × 25) + hidden_bonus + random_bonus
```

- `cat_score`: 역할 선호 카테고리 일치 시 1.0, 아니면 0.8 (보조 카테고리 일치 시 최대 1.0)
- `dist_score`: `exp(-0.0003 × distance_meters)` (가까울수록 1에 가깝게)
- `hidden_bonus`: 레벨 6 이상이고 히든젬이면 15
- `random_bonus`: 0~5 랜덤  
→ **역할(카테고리) + 거리 + 히든젬 + 랜덤**이 반영되고, **점수 순 상위 3개**가 나옵니다.

## 4. 역할(예술가·미식가·모험가 등)이 어떻게 쓰이나

### DB 경로

- **반경만** 역할별로 다름 (예: explorer 3000m, healer 800m, artist 2500m, foodie 1500m, challenger 4000m).  
  하지만 위에서 말한 대로 **REST 조회가 반경을 적용하지 않아** 사실상 반경은 무시됨.
- **점수/선택 로직**에는 역할이 **전혀** 들어가지 않음 (카테고리 40점 고정, 랜덤 3개).

### Mock 경로

- `ROLE_CATEGORY_MAP`으로 역할별 선호 카테고리 정의:
  - explorer: 이색장소, 히든스팟, 골목길, 빈티지샵
  - healer: 공원, 북카페, 숲, 사찰/교회, 도서관
  - archivist: 갤러리, 전시관, 뷰맛집, 건축물, 카페
  - relation: 맛집, 카페, 액티비티, 이색장소
  - achiever: 러닝코스, 헬스장, 챌린지스팟, 스터디카페
- **artist, foodie, challenger**는 Mock의 `ROLE_CATEGORY_MAP`에 없어서 기본 리스트 `[]`로 처리되며, 그러면 모든 장소가 `cat_score = 0.8`로 동일하게 취급됩니다.

정리: **DB 경로에서는 역할이 “반경 설정”에만 쓰이고(그마저도 미적용), Mock에서는 “카테고리 선호도 점수”로만 쓰입니다.**

---

## 5. 기분(Mood)이 어떻게 반영되나

- **장소 선택·점수 계산에는 기분이 전혀 쓰이지 않습니다.**
- `mood`는 **AI 서사(이야기) 생성**에만 전달됩니다.
  - `services/narrative_generator.py`  
    `generate_narratives_batch(places, role_type, user_mood=user_mood)`
  - 프롬프트에 `사용자는 지금 '{user_mood}' 기분입니다.` 형태로만 들어가서, **텍스트 톤/분위기**에만 영향을 줍니다.

즉, **어떤 3곳이 뽑히는지, 점수가 어떻게 붙는지와는 무관**합니다.

## 6. 계산식 요약 (쉽게)

### DB 경로 (현재)

- **반경**: 역할별 값 있음 → 하지만 **REST에서 미적용** → 전국 50개 수준 후보.
- **선택**: 후보 중 **랜덤 3개** (거리·점수 무관).
- **점수(표시)**:
  - `score = 85 + (평점×2) + max(0, 10 - 거리_km)`
  - 역할·기분은 점수/선택에 없음.

### Mock 경로

- **점수**  
  `점수 = (역할_카테고리_점수 × 40) + (거리_점수 × 25) + 히든보너스 + 0~5 랜덤`  
  - 거리_점수 = `exp(-0.0003 × 거리m)` (가까울수록 높음)
- **선택**: 점수 순 정렬 후 **상위 3개**.
- **기분**: 서사 문구에만 사용, 선택/점수에는 미사용.

---

## 7. 개선 방향 제안

1. **DB 경로**
   - `get_places_nearby` 결과를 **Python에서 Haversine으로 거리 필터** (역할별 `radius_meters` 이내만 유지).
   - 3개 선택을 **랜덤** → **점수(및 거리) 기준 정렬 후 상위 3개**로 변경.
   - 선택적으로, 역할별 선호 카테고리를 DB 경로에도 넣어 `score`/`score_breakdown`에 반영.
2. **기분**
   - 기분을 “서사만”이 아니라, 예를 들어 **카테고리/태그 가중치**나 **간단한 스코어 보정**에 넣으면, “지금 기분에 맞는 장소”가 더 잘 드러납니다.
3. **역할**
   - DB 경로에도 `ROLE_CATEGORY_MAP`(또는 DB 테이블)를 써서 `cat_score`를 넣고, **점수 = 기본 + 카테고리 + 거리 + 평점** 형태로 통일하면, “예술가/미식가/모험가” 선택이 추천 결과에 직접 반영됩니다.

이 문서는 `backend/routes/recommendations.py`, `backend/db/rest_helpers.py`, `backend/mock/mock_data.py`, `backend/services/narrative_generator.py` 기준으로 정리했습니다.
