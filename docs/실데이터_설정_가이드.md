# WhereHere 실데이터 설정 가이드 (Mock 없음)

## 확인한 내용 요약

- **백엔드**: `GET /api/v1/visits/{user_id}` → **visits** 테이블 조회  
- **백엔드**: `POST /api/v1/visits` → **visits** 테이블에 INSERT  
- **장소 조회**: `get_place_by_id(place_id)` → **places** 테이블에서 `id = place_id` (TEXT, 예: `kakao-26338954`)  
- **수집 스크립트**: `scripts/collect_simple.py`, `scripts/collect_kakao_places.py` → Supabase **places** 테이블에 INSERT (id, name, address, latitude, longitude, primary_category 등)

SQL 에디터에 있던 스크립트들 중에는 **visits**와 **user_visits**가 섞여 있고, **places**도 `place_id UUID` / `id TEXT` / `location GEOGRAPHY` 등 스키마가 달랐습니다.  
실데이터만 쓰려면 **한 번에 하나의 스키마**로 맞추는 것이 안전합니다.

---

## 1단계: Supabase에서 실데이터용 스키마 적용

1. Supabase 대시보드 → **SQL Editor** 이동  
2. 아래 파일 내용을 **전부 복사**해서 붙여넣고 **Run** 실행  
   - 파일: **`supabase/migrations/REAL_DATA_SCHEMA.sql`**

이 스크립트는 다음만 수행합니다.

- **places**  
  - `id` TEXT (예: `kakao-123`), `name`, `address`, `latitude`, `longitude`, `primary_category`, `secondary_categories`, `vibe_tags`, `description`, `average_rating`, `review_count`, `is_hidden_gem`, `typical_crowd_level`, `average_price`, `price_tier`, `is_active`  
  - 수집 스크립트가 넣는 형식과 동일
- **visits**  
  - `id` UUID, `user_id` TEXT, `place_id` TEXT, `visited_at`, `duration_minutes`, `rating`, `mood`, `spent_amount`, `companions`, `xp_earned`  
  - 앱이 방문 기록 조회/생성할 때 사용하는 테이블

기존에 **places**, **visits**를 쓰고 있었다면 이 스크립트가 **DROP 후 재생성**하므로, 기존 데이터는 사라집니다.  
다른 스키마(예: user_visits, 완전 새로 시작 스크립트의 places 등)와 혼용하지 말고, **이 스키마 하나만** 쓰는 것을 권장합니다.

## 2단계: 장소 데이터 수집 (Kakao → Supabase places)

Mock이 아닌 **실제 장소**를 쓰려면, Kakao Local API로 수집한 뒤 **places** 테이블에 넣어야 합니다.

### 방법 A: `scripts/collect_simple.py` (권장)

- 서울 25개 구 + 일부 수도권, 카페/음식점/술집 등  
- 수십~수백 개 단위로 수집 가능 (지역·카테고리 조합에 따라)

실행 전 (키 설정):

- **수집 스크립트는 로컬 PC에서 실행**합니다. Vercel/Railway 환경변수는 **앱·API 배포용**이라 스크립트와는 별도입니다.
- 아래 둘 중 하나만 하면 됩니다.
  - **옵션 1**: 프로젝트 루트에서 터미널로 환경변수 설정 후 실행 (`.env` 파일 없이 가능)
  - **옵션 2**: `backend/.env` 파일을 로컬에만 만들고 (Git 제외) `KAKAO_REST_API_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` 넣기

**옵션 1 – 터미널에서 한 번에 실행 (PowerShell 예시):**

```powershell
cd c:\Users\SAMSUNG\wherehere
$env:KAKAO_REST_API_KEY="여기에_카카오_REST_API_키"
$env:SUPABASE_URL="https://rftsnaoexvgjlhhfbsyt.supabase.co"
$env:SUPABASE_SERVICE_ROLE_KEY="여기에_서비스_롤_키"
python scripts/collect_simple.py
```

**옵션 2 – `backend/.env` 사용:**  
해당 파일에 위 세 변수만 넣고 `python scripts/collect_simple.py` 실행하면 됩니다.

- 스크립트가 Supabase **places** 테이블에 바로 INSERT 합니다.  
- `REAL_DATA_SCHEMA.sql`로 만든 **places** 스키마와 호환됩니다.

### 방법 B: `scripts/collect_kakao_places.py`

- 서울 25개 구 + 카페/음식점/술집/공원/문화시설/편의점  
- 지역·카테고리당 여러 페이지 수집으로, 더 많이 쌓을 수 있음

위와 같이 터미널 환경변수 또는 `backend/.env` 설정 후:

```powershell
python scripts/collect_kakao_places.py
```

- 역시 **places** 테이블에 INSERT합니다.

### 천 개 단위로 쓰고 싶을 때

- `collect_simple.py`: REGIONS / CATEGORIES 를 늘리거나, `search_places`의 `page`를 더 돌리면 됨  
- `collect_kakao_places.py`: `collect_region_category`에서 `page` 범위를 1~5 등으로 넓히면 됨  
- 두 스크립트 모두 **같은 places 스키마**를 사용하므로, 먼저 한 스크립트로 대량 수집한 뒤 다른 스크립트로 추가 수집해도 됩니다.

## 3단계: 방문(visits) 데이터

- **visits** 테이블은 **이미 앱에서 자동으로** 채워집니다. (구현 완료)  
  - 사용자가 퀘스트 완료 → “완료하고 XP 받기” → `POST /api/v1/visits` 호출 → **visits**에 한 건 INSERT  
- 따라서 **샘플 방문**을 넣고 싶다면,  
  - 먼저 2단계로 **places**를 채운 뒤,  
  - `place_id`를 실제 **places.id** 값(예: `kakao-26338954`)으로 해서 INSERT하면 됩니다.

예시 (Supabase SQL Editor에서 실행):

```sql
-- places에 데이터가 있는지 확인
SELECT id, name FROM places LIMIT 5;

-- 원하는 place_id 3개로 샘플 방문 삽입 (아래 place_id는 위 조회 결과로 교체)
INSERT INTO visits (user_id, place_id, visited_at, duration_minutes, rating, mood, xp_earned)
VALUES
  ('user-demo-001', 'kakao-26338954', NOW() - INTERVAL '1 day', 85, 4.8, '호기심', 150),
  ('user-demo-001', 'kakao-491234814', NOW() - INTERVAL '2 days', 60, 4.7, '행복', 120),
  ('user-demo-001', 'kakao-27172878', NOW() - INTERVAL '3 days', 45, 4.9, '평온', 100);
```

- `place_id`는 반드시 **places 테이블에 있는 id**로 넣어야 나의 지도에서 좌표가 표시됩니다. (없으면 기본 서울 좌표로 표시됨)

## 4단계: 앱이 실데이터만 쓰는지 확인

- **백엔드**  
  - `backend/db/rest_helpers.py`  
    - 방문 조회/생성: **visits** 테이블만 사용  
    - 장소 조회: **places** 테이블의 `id`(TEXT)로 조회  
  - Supabase 연결 시 Mock이 아니라 위 테이블을 그대로 사용합니다.  
- **프론트**  
  - 나의 지도: `GET /api/v1/visits/{user_id}` → **visits** + **places** 조인 결과(백엔드가 보강)로 표시  
  - 추천: **places** 테이블 기준으로 동작  

정리하면, Supabase에는 **REAL_DATA_SCHEMA.sql** 한 번 적용한 뒤, 장소는 **수집 스크립트**로만 채우고, 방문은 **앱(또는 위와 같은 SQL)**으로만 채우면 **Mock 없이 실데이터만** 사용할 수 있습니다.

## 요약 체크리스트

| 단계 | 작업 | 비고 |
|------|------|------|
| 1 | Supabase SQL Editor에서 `REAL_DATA_SCHEMA.sql` 실행 | places, visits 테이블 통일 ✅ |
| 2 | 수집 스크립트 실행 시 키 필요: 터미널 환경변수 또는 로컬 `backend/.env` | Vercel/Railway env는 앱용, 스크립트는 로컬 전용 |
| 3 | `python scripts/collect_simple.py` 또는 `collect_kakao_places.py` 실행 | **places 실데이터 수집·적재** (Kakao API → Supabase) |
| 4 | (선택) visits는 앱에서 퀘스트 완료 시 자동 저장됨. 샘플만 넣고 싶으면 SQL로 직접 INSERT | 나의 지도/통계용 |
| 5 | 백엔드/프론트 실행 후 추천·나의 지도 동작 확인 | Mock 없이 실데이터만 사용 |

이전에 “한 천 개 장소 불러왔던” 것은 위 2단계 수집 스크립트를 더 넓은 지역/카테고리/페이지로 돌리면 동일한 방식으로 다시 쌓을 수 있습니다.
