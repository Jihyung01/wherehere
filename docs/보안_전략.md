# WhereHere 보안 전략

> API·프론트·배포 환경에 적용할 보안 정책과 체크리스트입니다.

---

## 1. 환경 변수·비밀 관리

| 항목 | 규칙 |
|------|------|
| **백엔드** | `.env` 사용, **절대 커밋 금지**. `.env.example`만 저장하고 실제 값은 로컬/배포 환경에 설정. |
| **프론트** | `NEXT_PUBLIC_*` 만 클라이언트 노출. API 키·시크릿은 백엔드 전용. |
| **Vercel** | Project Settings → Environment Variables에 `NEXT_PUBLIC_API_URL`, `NEXT_PUBLIC_SUPABASE_*` 등 설정. |
| **Railway** | 백엔드 `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `SECRET_KEY` 등 설정. |

- Supabase **service role key**는 백엔드에서만 사용하고, 프론트에는 **anon key**만 사용.
- 카카오/구글 OAuth **클라이언트 시크릿**은 백엔드 또는 Supabase Auth 설정에만 둠.

---

## 2. CORS

- **개발**: `ALLOWED_ORIGINS`에 `http://localhost:3000` 등 필요한 포트 포함.
- **프로덕션**: 백엔드 `core/config.py`의 `ALLOWED_ORIGINS`에 실제 프론트 도메인만 등록 권장 (예: `https://wherehere-seven.vercel.app`).
- 현재 백엔드 `main.py`는 `allow_origins=["*"]`로 두었으나, **프로덕션에서는** `settings.ALLOWED_ORIGINS`를 사용해 출처 제한을 권장합니다.

---

## 3. API·입력 검증

- 모든 API 요청은 **Pydantic** 모델로 검증 (이미 적용됨).
- 파일 업로드: **크기 제한**(예: 4MB), **Content-Type** 화이트리스트(JPEG/PNG/GIF/WebP).
- 클라이언트에서 이미지 압축 후 업로드해 body 크기 초과(413) 방지.

---

## 4. 인증·세션

- **Supabase Auth**: 로그인 시 세션/쿠키 관리. Route Handler·Server Component에서는 `createServerClient`로 쿠키 기반 세션 사용.
- **백엔드 API**: 현재는 `user_id` 쿼리/바디로 사용자 구분. 추후 JWT 또는 세션 검증 도입 시 `core/security.py` 등에서 검증.
- 프로필·방문·소셜 등 사용자별 데이터는 **항상 user_id 기준**으로 필터링.

---

## 5. 프록시·405·CORS 방지

- 프론트가 **동일 출처**로 `/api/v1/*` 호출 시, Next.js API 라우트(`app/api/v1/[...path]/route.ts`)가 백엔드로 **프록시**.
- 효과: 브라우저 CORS preflight·405 이슈 감소, 백엔드 URL을 클라이언트에 노출하지 않아도 됨.
- Vercel 환경 변수 `NEXT_PUBLIC_API_URL`에 백엔드 URL(Railway 등) 설정 필요.

---

## 6. 에러 처리·사용자 피드백

- API 실패 시 **빈 catch** 대신 사용자에게 **토스트/알림**으로 메시지 표시.
- 네트워크 오류·413·503 등은 “다시 시도해 주세요” 또는 구체 메시지로 안내.

- [ ] `.env`가 Git에 포함되지 않았는지 확인 (`.gitignore`에 있음).

- [ ] 프로덕션 CORS: 백엔드 `ALLOWED_ORIGINS`에 실제 프론트 도메인만 등록했는지 확인 (선택).
- [ ] Supabase RLS 정책이 적용된 테이블은 적절한 범위로 제한되었는지 확인.
- [ ] 파일 업로드: 크기·타입 제한과 클라이언트 압축이 적용되었는지 확인.
